# Case Study: 从"想做个 AI 循环"到 Ralph Harness 的完整推导过程

> 本文档是 how-to-harness skill 的**真实案例教材**。用户 bellchen 带着一个模糊的想法"基于 OpenAI Harness Engineering 给我的排障 AI Agent 做个自优化循环"进来，通过 9 轮对话推导出了完整的 Ralph Harness 系统方案。

## 0. 初始状态

**用户提供的素材**：
- 方法论锚点：OpenAI Harness Engineering
- 概念：Ralph Wiggum 循环、"Humans steer, agents execute"
- 现有资产：2 个项目 + 1 个数据仓库
- 模糊需求："给排障 AI Agent 做个自优化循环"

**Agent 的第一反应错误示范**：立即输出完整架构图、开始写 PRD、提供技术栈建议。

**Agent 的正确第一反应**：先做 Layer 1 Capture Context。

---

## 1. Layer 1：Capture Context 的执行

Agent 第一轮做了两件事：

### 1.1 读完用户提供的 3 个项目源码

并行读了：
- `ticket-analyzer-skill`（Analyzer 能力）
- `diagnose-skill-review`（Reviewer 能力）
- `ticket-analyzer-output`（数据事实源）

**关键发现**：用户已经有了 Analyzer + Reviewer，但**两者之间的闭环还没打通**。

### 1.2 向用户复述理解

> "我听到的是这样的：您的 Analyzer 在沉淀 Skill、Reviewer 在评估 anclaw，但两者之间缺少一个自驱的循环来让 Skill 持续演化。您说的 'Ralph 循环' 其实就是要把这个闭环补上。对吗？"

用户确认："对。"

**教训**：复述是最便宜的"验证理解"手段，花 30 秒换避免一整轮歧义。

---

## 2. Layer 3：Socratic 追问的 9 轮实战

### Round 1: 多子目标优先级

**问题**：
```
A. 全景蓝图（把终局画出来）
B. MVP 闭环（先让循环转起来）
C. 管理赋能（周报 + Dashboard）
D. 评测体系（Gold Set + Harness）
```
**用户答**：`DBAC`（一次 4 个字母排完全序）

**Agent 内心活动**：用户把 D 放第一，说明他已经内化了 Harness Engineering 的"Eval is foundation"。后续所有设计要围绕 D 优先。

### Round 2: 评测消费者排序

**问题**：
```
R. Ralph 自身（循环自驱）
A. 管理者 bellchen
B. Owner（产品线负责人）
C. 客服（终端用户）
```
**用户答**：`RABC`

**Agent 内心活动**：用户选 R 第一，说明他真的信"Humans steer, agents execute"——系统首先为 Agent 自己服务，人类是掌舵者。这是一个极强的方法论对齐信号。

### Round 3: 评测策略

**问题**：
```
A. 金标准集（Gold Set）— 软评估
B. 回归集（硬门禁）— Skill 变更必须过
C. 影子对比（并行跑新旧版本）
D. 人工抽检
```
**用户答**：B（回归集硬门禁）

**Agent 内心活动**：硬门禁意味着"宁可堵住也不让坏东西上线"。这是 CI/CD 级别的严格性，后续方案的自动化程度可以做得很激进。

### Round 4: 人类闸门 + 投入时间

**问题**（一次 3 个子问题）：
- 4.1 Owner 审批哪些内容？
- 4.2 管理者每周投入多少？
- 4.3 团队资源？

**用户答**：`3,4 是人工` + `2-3 小时/周` + `AI coding agent 辅助开发部署`

**Agent 内心活动**：用户只投入 2-3h/周 + 没有工程师团队，**这个约束极其严**。后续方案必须满足：
- 所有组件是 Agent-friendly（YAML config / Git PR，不能依赖 GUI）
- 人类介入频率必须极低
- 开发与运维全部 AI-agent 可完成

**教训**：这一轮是关键——**资源约束决定了方案的形态**。如果用户说"我有 3 个工程师"，方案可以做得重；说"只有我自己 + AI agent"，方案必须做得极轻。

### Round 5: Lite Improver 档位

**问题**：
```
L1. 只改措辞
L2. 改结构（含拆分/合并）
L3. 改能力（新工具 / 新分支）
```
**用户答**：L1 + L2，担心"全人工介入耗费太多人力成本"

**Agent 的应对**：发现用户逻辑不自洽的担忧，主动扮演**魔鬼代言人**：
> "L2 自动改结构是有风险的——它可能越改越偏。建议先 L1，看清楚失败模式再开 L2。"

用户坚持 L2，理由是：**"ticket-analyzer-skill 本身就允许自动写、改、拆 skill"**。

**Agent 的调整**：用户说得对——**既然现有系统已经在生产做这件事，L2 风险并没有增加**。立即接受决策，但补上 3 层安全网：
- L2 禁区清单（红线 / 工具调用 / 元数据字段不能改）
- 跨 Skill 回归（改动影响邻居 Skill 时回滚）
- 7 天观察期 + 自动回滚

**教训**：魔鬼代言人的作用不是否决用户，而是**逼用户把理由说清楚**。用户能说清楚理由 → 接受决策 + 补安全网；说不清 → 帮用户想想是不是真的要这么做。

### Round 6: 熔断策略

**问题**：
```
A. 限频也熔断
B. 限频不熔断（靠现有重试机制）
C. 全自动恢复
D. 全手动恢复
```
**用户答**：`限频不熔断，复用现有机制` + `其他故障手动恢复`

**Agent 内心活动**：用户分清了"限频"和"真实故障"的区别——限频是常态不是故障。这是**对现有系统的尊重**，避免重复造轮子。

### Round 7-9: 其他维度

依次锁定：
- 红线规则（3 级分类 + 3 个来源）
- 仓库拓扑（3 现有 + 1 新建 ralph-harness）
- MVP 时间线（4 周 + 7 天观察）

---

## 3. Layer 4：分节呈现 + 逐节 Approve

**不一次性输出完整方案**，而是按 4 节：
1. 顶层架构
2. 组件详细设计 + MVP Scope
3. 执行语义 + 红线设计
4. 数据模型 + 仓库拓扑

每节结束必问"approve 还是调整？"

**用户在第 2 节的调整**：
- 把 Improver 从 MVP 排除 → 改为包含
- 把 Gold Set 从 80 个人工样本 → 改为规则自动提名
- L2 观察期从 3 天 → 改为 7 天

这些调整让最终方案质量显著提升。**如果一次性输出完整方案，用户可能懒得细看**。

---

## 4. Layer 4：三件套交付

全部 Approve 后，同时产出：
- Executive One-Pager
- Week 1 Kickoff 工作清单（6 配置 + 3 prompt 模板）
- 完整 PRD（按内化的 PRD Schema 产出，见 Layer 4.1）

---

## 5. 复盘：用户问"你用了什么框架"

Agent 诚实回答：
- Layer 1: Harness Engineering 作为方法论锚点
- Layer 2: 先问再写的纪律
- Layer 3: Socratic A/B/C/D 追问 + 决策锁定
- Layer 4: 内化的 PRD Schema 作为终点模板

**并主动暴露了 4 个不足**：
1. 第 4 轮塞了 3 个子问题（应该每轮 1 个）
2. Cost 维度太晚暴露（应该在 brainstorming 早期）
3. 没有引入反对派视角（后来在 L2 档位时补上了）
4. 擅自命名了 ralph-harness 仓库（应该给候选）

**教训**：复盘自己的方法论缺陷**增强用户信任**，而不是削弱。

---

## 核心模式总结（可复用到其他项目）

| 阶段 | 关键动作 | 避免 |
|------|---------|------|
| Capture | 读用户的现有资产 + 复述理解 | 凭空想象 |
| Socratic | A/B/C/D + 推荐 + 理由 | 开放式问题 |
| 决策锁定 | 每 3 轮复述一次 | 走着走着忘了 |
| 分节呈现 | 每节 Approve | 一次性甩方案 |
| 魔鬼代言人 | 质疑关键决策 | 完全讨好用户 |
| 交付 | 三件套 | 只交 PRD |
| 复盘 | 暴露自己的不足 | 装得完美 |

---

## 数字化的结果

这个案例最终的产出：
- **时长**：9 轮对话 + 4 节设计 = 约 2 小时用户时间
- **产出**：1 份 design doc + 1 份 PRD + 1 份 One-Pager + Week 1 Kickoff（6 configs + 3 prompts） + 配套的 git 仓库
- **决策锁定数**：20+ 条关键决策全部有明确归属
- **下一步明确性**：可直接给 AI coding agent 启动 Week 1

**如果没用这个框架**：可能要写 2-3 版 PRD 才能达到同样的清晰度，时间 3-5 倍。
