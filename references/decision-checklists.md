# Decision Checklists · Harness 循环类系统的硬性决策清单

> 这份清单帮助 Agent 在 Socratic 追问阶段**不遗漏关键维度**。**每一节都对齐 SKILL.md 的 Layer 0 Harness Principles（HP-1~HP-7）**——这些不是"问用户要不要"的问题，而是"必须有，问用户从哪来"的问题。
>
> **适用范围**：本 skill 专注于 Harness 闭环 / 自优化系统。

---

## 类型 A：AI Agent 自优化循环（Ralph / Harness 类）

典型场景：自动从数据中生成能力 + 自动评测 + 自动改进 + 人类掌舵

### A.0 方法论层（锚点与演化资产）
- [ ] 参考方法论锚点是什么？（OpenAI Harness / LangGraph / AutoGPT / ……）
- [ ] 循环演化的对象是什么？（能力资产 / 模型参数 / Prompt 库 / 知识库 / Skill 目录）
- [ ] 输入原料是什么？（工单 / 日志 / 用户反馈 / 代码提交）

### A.1 · 对齐 HP-1（Eval is foundation）· 评测层
**这不是"要不要评测"的问题，是"评测从哪来"的问题。**
- [ ] **Gold Set 从哪来**？（人工标注 / 规则自动提名 / 历史高质量样本）— 必选
- [ ] **冷启动规模多少**？（最低多少条才能启动循环？）
- [ ] 评测频率？（每次变更 / 每日 / 每周）
- [ ] 打分方式？（LLM-as-judge / 规则 / 人工 / 混合）
- [ ] 通过阈值？（硬门禁 / 软评估）
- [ ] LLM-judge 稳定性如何保证？（多次打分 + 方差阈值）
- [ ] 评测集更新机制？（如何避免过拟合评测集本身）

> **红线**：用户说"先上线，评测集以后再补"→ 拦住，这违反 HP-1。

### A.2 · 对齐 HP-2（Gates not code）· 治理闸门层
**这不是"需不需要人审"的问题，是"每级闸门的边界与升级路径是什么"的问题。**
- [ ] **闸门分几级**？（典型：L1 自动 / L2 Owner / L3 Manager / 红线熔断）— 必选
- [ ] **每级闸门的审批边界**？（什么类型的变更归哪一级）
- [ ] **每级闸门的超时策略**？（多久没人审自动做什么：拒绝 / 降级 / 自动过）
- [ ] **升级路径**？（未通过 → 升到哪个人工队列；再未通过 → 再升到哪）
- [ ] 审批入口？（企微 / 邮件 / Dashboard / CLI）
- [ ] 审计留痕机制？（谁在什么时间做了什么决定，能追溯多久）

> **红线**：任何让人类"亲自下场改代码 / 改 Prompt / 改数据"的决策→ 指出违反 HP-2，建议改为"加/调/审闸门"。

### A.3 · 对齐 HP-3（Idempotent & resumable）· 循环执行层
**这不是"要不要幂等"的问题，是"每一步如何幂等"的问题。**
- [ ] 循环的每一步是否**幂等**？（重跑一次结果一致）
- [ ] 中间态存哪？（断点文件 / DB / 对象存储）
- [ ] 来源数据是否有**稳定的 ID**？（避免同一条输入被当成两条）
- [ ] 断点恢复的机制？（从哪一步续跑 / 需要哪些上下文）
- [ ] 并发冲突如何避免？（锁 / 乐观并发 / 单例执行）

> **红线**：用户说"每次跑结果不一样也行"→ 拦住，这违反 HP-3，追问为什么不行（不可复现 / 无法调试 / 无法回滚）。

### A.4 · 对齐 HP-4（Small reversible steps）· 改进器层
**这不是"改进器能做什么"的问题，是"改进粒度 / 观察期 / 回滚条件"的问题。**
- [ ] **改进粒度**？（一条 Prompt / 一个子模块 / 整条 Skill —— 越小越好）
- [ ] **观察期多长**？（改完立即生效 vs 灰度 N 天再全量）
- [ ] **自动回滚的触发条件**？（评测分跌 X% / 红线事件 / 人工一键）
- [ ] 改进器档位？（L1 改措辞 / L2 改结构 / L3 改能力）
- [ ] 最多尝试次数？
- [ ] 禁区清单？（哪些字段 / 文件 / 模块不允许自动改）

> **红线**：用户说"改完直接上线"→ 指出违反 HP-4，讨论弱一致性 / 影响爆炸半径。

### A.5 · 对齐 HP-5（Automation tiers with escalation）· 分级自动化
**这不是"要不要自动化"的问题，是"自动化分几档 + 档间如何升级"的问题。**
- [ ] **有几级自动化**？（典型：L1 全自动 / L2 Owner 必审 / L3 Manager 必审）
- [ ] **每级的权限边界**？（哪些动作归 L1，哪些必须升到 L2）
- [ ] **升级触发条件**？（哪些事件/异常/阈值触发从 L1 升到 L2/L3）
- [ ] **降档触发条件**？（什么情况允许从 L3 降到 L2，避免永久升档）

### A.6 · 对齐 HP-6（Asset versioning）· 资产版本化
**这不是"要不要版本化"的问题，是"版本化策略是什么"的问题。**
- [ ] **演化资产用什么版本化**？（git / 数据库版本表 / 对象存储快照）
- [ ] 版本元数据要存什么？（时间 / 改动者 / 触发事件 / 前后 diff）
- [ ] 保留多久？（90 天 / 6 个月 / 永久）
- [ ] 一致性约束？（Skill / Prompt / Knowledge 的版本是否要对齐）
- [ ] 回滚粒度？（单资产 / 整个快照）

### A.7 · 对齐 HP-7（Human time budget）· 人类参与预算
**这不是"要不要人类参与"的问题，是"先把人类时间预算列出来，再看能支撑多少自动化"的问题。**
- [ ] **管理者每周可投入小时数**？— **先列预算**
- [ ] **Owner 每天可投入小时数**？— **先列预算**
- [ ] 标注人力 / 审批人力？— **先列预算**
- [ ] 预期 ESCALATE 次数 / 周？
- [ ] 预期红线事件次数 / 周？
- [ ] **算一下：当前设计下预算够吗**？（不够则**必须回头降自动化 / 提高闸门阈值 / 引入更多角色**，不能默认"人不够就加班"）
- [ ] 周报 / 日报频率？
- [ ] 告警渠道？（企微 / 邮件 / Dashboard）

### A.8 红线层（系统级安全边界）
- [ ] 红线有几级？（critical / warning / behavior）
- [ ] 红线来源？（人工定义 / 自动挖掘 / 人工录入）
- [ ] 红线命中动作？（熔断 / 告警 / 降级）
- [ ] 红线治理权限？（谁能加 / 谁能删 / 谁能豁免）

### A.9 熔断层
- [ ] 哪些故障需要熔断？
- [ ] 哪些故障不熔断（交给现有重试机制）？
- [ ] 恢复方式？（自动 / 手动）
- [ ] 熔断粒度？（单个组件 / 整个循环）

### A.10 交付层
- [ ] MVP 范围？
- [ ] MVP 时间线？
- [ ] 验收标志（可观测、二元判定）？
- [ ] V1.1 / V2 路线？

## 类型 B：评测驱动的改进系统（Evaluation-Driven，无完整闭环）

典型场景：已有 AI 系统，想加一层评测 + 改进回路（但改进仍由人触发）

**注意**：这类场景大部分维度与类型 A 重叠，但由于**改进不自动**，HP-3/HP-4/HP-5 的强制程度略低。核心仍是 HP-1 和 HP-7。

### B.1 评测设计（同 A.1，对齐 HP-1）
- [ ] 评测什么？（准确率 / 覆盖率 / 时延 / 成本 / 用户满意度）
- [ ] 评测集规模？（冷启动数量 / 稳态数量）
- [ ] 评测集更新频率？
- [ ] 评测集与线上数据的分布一致性如何保证？

### B.2 改进触发
- [ ] 什么情况触发改进？（评测分下降 / 达到周期 / 人工发起）
- [ ] 改进由谁执行？（AI agent / 工程师 / 混合）
- [ ] 改进有边界吗？
- [ ] 改进后的影响评估机制？

### B.3 对照组
- [ ] 是否需要 A/B 测试？
- [ ] 影子对比（shadow run）？
- [ ] 基线版本如何保留？（与 HP-6 对齐）

### B.4 人类预算（同 A.7，对齐 HP-7）
- [ ] 改进工程师每周投入？
- [ ] 评测标注人力？
- [ ] 评审会议频率？

## 类型 C：人机协作的治理系统（Human-in-the-Loop Governance）

典型场景：AI 做决策，但关键点必须人工审批。**这类场景 HP-2 是绝对核心**。

### C.1 · 对齐 HP-2 · 角色与权限
- [ ] 有几类角色？（auto / owner / manager / admin）
- [ ] 角色之间的权限边界？
- [ ] 同一人可以身兼多角色吗？如何避免"自审自批"？

### C.2 · 对齐 HP-2 · 审批矩阵
- [ ] 什么类型的变更走哪类角色？
- [ ] 自动合并的边界在哪？
- [ ] 升级路径？
- [ ] 超时策略？

### C.3 · 对齐 HP-6 · 通知与追溯
- [ ] 审批请求如何送达？
- [ ] 审批结果如何记录？
- [ ] 审计链路如何建立？保存多久？

### C.4 紧急情况
- [ ] 有紧急审批通道吗？
- [ ] 绕过审批的条件是什么？（绕过一定要留痕并事后复盘）
- [ ] 红线事件处理流程？

### C.5 · 对齐 HP-7 · 人类预算
- [ ] 每类审批人每周可投入小时数？
- [ ] 预期审批数量 / 周？预算够吗？

## 通用维度（所有类型都要考虑）

### 成本维度（最容易被忽略！）
- [ ] LLM API 单日成本预估？
- [ ] 人力成本（审批 / 维护 / 标注）？
- [ ] 基础设施成本？
- [ ] 开发时间成本？
- [ ] 预期 ROI / 回收周期？

### 风险维度
- [ ] 技术风险（限流 / 不稳定 / 依赖故障）？
- [ ] 业务风险（错误建议 / 误导用户）？
- [ ] 合规风险（数据脱敏 / 隐私）？
- [ ] 组织风险（Owner 变动 / 团队调整）？

### 命名维度（避免擅自决定！）
- [ ] 系统名？
- [ ] 仓库名？
- [ ] 核心组件名？
- [ ] 关键抽象概念名？

**规则**：每一个命名都应该**给用户 2-3 个候选让用户选**，不要擅自决定。

---

## 使用本清单的方式

1. 对照用户的系统类型选 A/B/C
2. **先过 SKILL.md 的 Layer 0 验收 7 问**，不过不进入后续追问
3. 按 A.0/A.1/A.2... 顺序逐块问（不跳序，依赖拓扑严格）
4. 每一块问完做一次小复述："这块我们锁定了 X/Y/Z 三个决策，对齐的是 HP-N"
5. 全部问完后合并通用维度（成本 / 风险 / 命名）
6. 确认无遗漏且 **所有 HP-1~HP-7 均已有具体答案**后，进入方案分节呈现阶段

## 反面教材：不该直接问的问题

| ❌ 别这样问 | ✅ 应该怎么问 |
|------------|--------------|
| "你需要 Gold Set 吗？" | "Gold Set 必须有（HP-1）。它从哪来？A. 人工标注 B. 规则自动提名 C. 历史高质量样本。我建议..." |
| "评测策略怎么搞？" | "评测策略有 4 个候选：A. Gold Set 硬门禁 B. 软评估 C. 影子对比 D. 无评测。无评测违反 HP-1 不可选。我建议 A，因为..." |
| "熔断怎么设计？" | "熔断触发有 5 种情况：... 您希望其中哪些熔断？同时这和 HP-5 的升级路径挂钩，我们一并定。" |
| "要不要版本化？" | "演化资产必须版本化（HP-6）。用 A. git B. DB 版本表 C. 对象存储快照。我建议..." |
| "管理者每周投多少？" | "我们先列预算（HP-7）：管理者每周 N 小时、Owner 每天 N 小时，再看当前设计下预期审批数会不会超预算。" |
| "你希望用什么技术栈？" | 这不是决策，是实现。除非有硬约束（比如必须用 Python），否则不要提前问 |
| "架构怎么画？" | 架构不该先问——应该等决策都锁定后自然浮现 |
