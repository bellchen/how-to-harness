# Decision Checklists · 不同系统类型的典型决策清单

> 这份清单帮助 Agent 在 Socratic 追问阶段**不遗漏关键维度**。按用户要设计的系统类型，选对应的清单逐条过。

---

## 类型 A：AI Agent 自优化循环（Ralph / Harness 类）

典型场景：自动从数据中生成能力 + 自动评测 + 自动改进 + 人类掌舵

### A.1 方法论层
- [ ] 有没有参考方法论锚点？（OpenAI Harness / LangGraph / AutoGPT / ...）
- [ ] 循环演化的对象是什么？（能力资产 / 模型参数 / Prompt 库 / 知识库）
- [ ] 输入原料是什么？（工单 / 日志 / 用户反馈 / 代码提交）

### A.2 评测层
- [ ] Gold Set 从哪里来？（人工标注 / 规则自动提名 / 历史高质量样本）
- [ ] 评测频率？（每次变更 / 每日 / 每周）
- [ ] 打分方式？（LLM-as-judge / 规则 / 人工 / 混合）
- [ ] 通过阈值？（硬门禁 / 软评估）
- [ ] LLM-judge 稳定性如何保证？（多次打分 + 方差阈值）

### A.3 改进层
- [ ] 改进器能做什么？（L1 改措辞 / L2 改结构 / L3 改能力）
- [ ] 最多尝试次数？
- [ ] 禁区清单？（哪些字段不能改）
- [ ] 改进后的观察期？
- [ ] 自动回滚的触发条件？

### A.4 治理层
- [ ] 有几级闸门？（auto / owner / manager）
- [ ] 每级闸门的审批边界？
- [ ] 超时策略？（多久没人审自动做什么）
- [ ] ESCALATE 路径？（升级到哪个人工队列）

### A.5 红线层
- [ ] 红线有几级？（critical / warning / behavior）
- [ ] 红线来源？（人工定义 / 自动挖掘 / 人工录入）
- [ ] 红线命中动作？（熔断 / 告警 / 降级）
- [ ] 红线治理权限？（谁能加 / 谁能删 / 谁能豁免）

### A.6 熔断层
- [ ] 哪些故障需要熔断？
- [ ] 哪些故障不熔断（交给现有重试机制）？
- [ ] 恢复方式？（自动 / 手动）
- [ ] 熔断粒度？（单个组件 / 整个循环）

### A.7 人类参与层
- [ ] 管理者每周投入多少？
- [ ] Owner 每天投入多少？
- [ ] 终端用户是否需感知？
- [ ] 周报 / 日报频率？
- [ ] 告警渠道？（企微 / 邮件 / Dashboard）

### A.8 数据层
- [ ] 数据事实源在哪？（单一 truth）
- [ ] 版本化策略？（git / 数据库版本 / 快照）
- [ ] 历史归档策略？

### A.9 交付层
- [ ] MVP 范围？
- [ ] MVP 时间线？
- [ ] 验收标志？
- [ ] V1.1 / V2 路线？

## 类型 B：评测驱动的改进系统（Evaluation-Driven）

典型场景：已有 AI 系统，想加一层评测 + 改进回路

### B.1 评测设计
- [ ] 评测什么？（准确率 / 覆盖率 / 时延 / 成本 / 用户满意度）
- [ ] 评测集规模？（冷启动数量 / 稳态数量）
- [ ] 评测集更新频率？
- [ ] 评测集与线上数据的分布一致性如何保证？

### B.2 改进触发
- [ ] 什么情况触发改进？（评测分下降 / 达到周期 / 人工发起）
- [ ] 改进由谁执行？（AI agent / 工程师 / 混合）
- [ ] 改进有边界吗？

### B.3 对照组
- [ ] 是否需要 A/B 测试？
- [ ] 影子对比（shadow run）？
- [ ] 基线版本如何保留？

---

## 类型 C：人机协作的治理系统（Human-in-the-Loop Governance）

典型场景：AI 做决策，但关键点必须人工审批

### C.1 角色定义
- [ ] 有几类角色？
- [ ] 角色之间的权限边界？
- [ ] 同一人可以身兼多角色吗？角色边界如何保持？

### C.2 审批矩阵
- [ ] 什么类型的变更走哪类角色？
- [ ] 自动合并的边界在哪？
- [ ] 升级路径？

### C.3 通知与追溯
- [ ] 审批请求如何送达？
- [ ] 审批结果如何记录？
- [ ] 审计链路如何建立？

### C.4 紧急情况
- [ ] 有紧急审批通道吗？
- [ ] 绕过审批的条件是什么？
- [ ] 红线事件处理流程？

---

## 类型 D：普通软件产品 / 功能（非 AI）

典型场景：常规的产品 PRD。**这种情况可以按 `references/deliverables.md` 的 PRD 模板直接写，不需要走完整的 Socratic 流程，但仍然可以复用本 skill 的决策锁定纪律**。

但如果需要额外的决策锁定：
- [ ] MVP vs 完整版的边界
- [ ] 数据模型的边界
- [ ] 和现有系统的集成点
- [ ] 用户迁移策略
- [ ] 性能 / 成本预算

---

## 通用维度（所有类型都要考虑）

### 成本维度（最容易被忽略！）
- [ ] LLM API 单日成本预估？
- [ ] 人力成本（审批 / 维护）？
- [ ] 基础设施成本？
- [ ] 开发时间成本？
- [ ] 预期 ROI / 回收周期？

### 风险维度
- [ ] 技术风险（限流 / 不稳定 / 依赖故障）？
- [ ] 业务风险（错误建议 / 误导用户）？
- [ ] 合规风险（数据脱敏 / 隐私）？
- [ ] 组织风险（Owner 变动 / 团队调整）？

### 命名维度（避免擅自决定！）
- [ ] 系统名？
- [ ] 仓库名？
- [ ] 核心组件名？
- [ ] 关键抽象概念名？

**规则**：每一个命名都应该**给用户 2-3 个候选让用户选**，不要擅自决定。

---

## 使用本清单的方式

1. 对照用户的系统类型选 A/B/C/D
2. 按 A.1/A.2/A.3... 顺序逐块问（不跳序）
3. 每一块问完做一次小复述："这块我们锁定了 X/Y/Z 三个决策"
4. 全部问完后合并通用维度（成本 / 风险 / 命名）
5. 确认无遗漏后进入方案分节呈现阶段

## 反面教材：不该直接问的问题

| ❌ 别这样问 | ✅ 应该怎么问 |
|------------|--------------|
| "你觉得应该怎么评测？" | "评测策略有 4 个候选：A. Gold Set 硬门禁 B. 软评估 C. 影子对比 D. 无评测。我建议 A，因为..." |
| "熔断怎么设计？" | "熔断触发有 5 种情况：... 您希望其中哪些熔断？" |
| "架构怎么画？" | 架构不该先问——应该等决策都锁定后自然浮现 |
| "你希望用什么技术栈？" | 这不是决策，是实现。除非有硬约束（比如必须用 Python），否则不要提前问 |
